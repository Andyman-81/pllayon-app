Add an injury and physical flag system to the Pllay On app.
Do not change any existing functionality.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
OVERVIEW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Three roles interact with this system:

 ATHLETE  → logs the injury or physical concern
 PARENT   → can also log on behalf of athlete + receives updates
 COACH    → receives the flag, responds with course of action

Flow:
 Athlete or Parent logs flag
   → Coach is notified (in-app banner)
   → Coach reviews and responds
   → Athlete and Parent see the response
   → Flag is resolved or escalated

This is not a medical system.
It is a communication and tracking tool.
Always include a disclaimer that serious injuries
require medical attention — not a coach response.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
DATABASE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

create table injury_flags (
 id uuid primary key default gen_random_uuid(),
 athlete_id uuid references athletes(id) on delete cascade,
 logged_by text not null, -- 'athlete' or 'parent'
 logged_by_name text,
 created_at timestamptz default now(),
 updated_at timestamptz default now(),

 -- Injury details
 body_area text not null,
 side text, -- 'left', 'right', 'both', 'centre', null
 concern_type text not null, -- 'pain', 'tightness', 'swelling', 'fatigue', 'other'
 severity int not null check (severity between 1 and 5),
 onset text not null, -- 'sudden', 'gradual'
 when_occurred text, -- free text e.g. "During Thursday S&C session"
 description text not null,
 affects_training boolean default true,
 week_number int,
 day_of_week text,

 -- Status
 status text default 'open',
 -- 'open', 'coach_reviewed', 'monitoring', 'cleared', 'referred'

 -- Coach response
 coach_response text,
 coach_action text,
 -- 'continue_modified', 'rest', 'reduce_load',
 -- 'physio_referral', 'monitor', 'cleared_to_train'
 coach_responded_at timestamptz,
 training_modification text,
 return_to_full_training text, -- estimated date or condition

 -- Follow-up
 follow_up_required boolean default false,
 follow_up_date text,
 resolved_at timestamptz
);

alter table injury_flags enable row level security;

create policy "injury_athlete_own" on injury_flags
 using (auth.uid() = athlete_id);

-- Coaches can read flags for their linked athletes
create policy "injury_coach_read" on injury_flags
 for select using (
   exists (
     select 1 from coach_athlete_links
     where coach_id = auth.uid()
     and athlete_id = injury_flags.athlete_id
   )
 );

-- Coaches can update (respond to) flags for their linked athletes
create policy "injury_coach_update" on injury_flags
 for update using (
   exists (
     select 1 from coach_athlete_links
     where coach_id = auth.uid()
     and athlete_id = injury_flags.athlete_id
   )
 );

DESIGN RULES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 - Red (#FF4936) is the signal colour throughout this feature
   It already means "Performance Expression / competition" in
   the phase system — in injury context it means urgency.
   Use sparingly and consistently.
 - Severity 1–2 flags: amber — low visual weight
 - Severity 3 flags: orange — medium visual weight
 - Severity 4–5 flags: red — maximum visual weight
 - Cleared flags: green — resolved, no concern
 - Space Mono for all labels, badges and status text
 - Barlow Condensed for page and section headers
 - Inter for all body text, descriptions and responses
 - No motivational language
 - Medical disclaimer always visible on the entry form
 - Mobile-first — all inputs and buttons minimum 44px touch target
 - Auto-save on all text fields (1500ms debounce)
